HOW DOES SSL WORK?

Exploring the cryptography and protocols behind SSL

SSL Certificates create trust in users and provide an encrypted connection secure from malicious attacks. Online businesses must establish trust in an ever increasingly competitive and dangerous online environment. With recent spates of large companies like Equifax leaking data, SSL encryption is an essential step towards securing client and server side data transmissions. 
So How Does It Work?

SSL Encryption is a multi-certificate cryptographic process using key pairs and a series of verifying certificates to ensure that end to end data transmissions are encrypted, secure, and authentic. 

This involves creating a Certificate Signing Request (CSR) on the target server, which creates a private and public key (usually 256-bit SSH or AES) that is stored on the server. To create an actual SSL certificate for your application or website, the generated CSR file has to be sent to a Certificate Authority (CA) that issues a SSL certificate for the app/site. the CA uses the CSR file and public key to create data structures that match the private key without compromising the private key to prying eyes.

Once the CA issues a SSL certificate, the certificate is then installed server side on the instance that is hosting the application or site. An intermediate certificate should also be installed to establish the credibility of the SSL certificate by tying it to a root certificate (the root certificate is usually from a high quality and trusted provider).


 

The digital signature of the certificate matters most browsers will have a store of Certificate Authorities that are trusted, this is called a Trusted Root CA store. If your SSL certificate is not signed by a trusted provider of SSL certificates, browsers could still potentially flag your site as insecure and prevent users from accessing the site. Particularly in the latest revisions of Google Chrome where any sites that are still operating on the older and unsecured HTTP protocol are automatically flagged as unsafe and blocked.

Paired private/public keys created with a Certificate Signing Request (server side) to hash information using a paired key schema. Preventing packet sniffers or malicious software from accessing raw data output or input

Authentication

Stop false flag attacks - Intermediate, Root and Server SSL Certificates verify the authenticity of client-side and server-side transmissions so data isn't erroneously sent to a mailicious 3rd party.

Data Integrity

Ensure the data submitted and requested by the server or client is the data that is delivered. Prevents man-in-the-middle attacks by hashing communications with a public key and decrypting with a private key.
